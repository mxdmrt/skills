---
name: write-design-brief
description: Turn clarified discovery into a design-facing brief. Use only when the user explicitly asks for a design brief, product brief, UX brief, or design handoff that preserves problem framing, user context, evidence, assumptions, constraints, exploration directions, and open questions.
---

Assume the user is choosing to move from investigation into design shaping. Do not decide that handoff moment for them.

Be concise, skeptical, and useful to designers. Do not hide weak discovery inside polished language. If the inputs are too vague, say what is missing before drafting. If only part of the brief can be drafted with confidence, draft that part and mark the rest as assumptions or open questions.

If a blocking input is missing, ask one sharp question at a time. Before drafting, check the biggest missing input first:
- What problem are we designing around?
- Who has it, in what context?
- What evidence supports this framing?
- Which assumptions are still doing too much work?
- What business, technical, policy, accessibility, or content constraints matter?
- What is explicitly out of scope?
- What design exploration is invited, and what is already decided?

When drafting, produce:
- Problem framing
- User context and jobs
- Evidence summary and confidence
- Assumptions and unknowns
- Constraints and non-goals
- Design principles or qualities to preserve
- Exploration directions
- Risks and edge cases
- Open questions
- Suggested next design step

Keep the brief design-facing, not engineering-task-facing. Do not convert it into acceptance criteria unless the user asks for a delivery issue instead.
